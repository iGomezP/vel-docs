## Decisions Made

### Documentation

VEL documentation must be stored in Git using Markdown and published through MkDocs.

Generated artifacts such as `site/` must not be committed.

### Printer Service

The Epson L4360 will use two paths:

```text
Primary family path:
Phones / Windows Devices
        ↓ WiFi
Epson L4360
```

```text
Fallback / lab-managed path:
Windows Devices
        ↓ Network / IPP
Fedora Server - CUPS
        ↓ USB
Epson L4360
```

Phones will use Epson WiFi directly to preserve:

- Epson mobile app support.
- Scanning.
- Ink level checks.
- Maintenance.
- Wake from app.

Fedora CUPS will be used as a stable USB fallback mainly for Windows clients.

### Security / Documentation Redaction

Public runbooks should avoid exposing:

- Internal IP addresses.
- Hardware serial numbers.
- Exact router models when unnecessary.
- Sensitive network identifiers.

Private operational values should be stored in local files excluded from Git.

---

## Lessons Learned

- Driverless printing is convenient but not always reliable.
- Official vendor drivers may still be required for stable USB printer workflows.
- CUPS works well as a Windows-accessible print server.
- Android/Mopria does not always accept CUPS printer URLs.
- Home and lab network separation is useful, but cross-network services require routing or controlled forwarding.
- Operational infrastructure work should be documented with runbooks.
- A good runbook should separate public procedure from private local values.
- Work items can be created after incident resolution when operational urgency comes first.
